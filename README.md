# Git과 GitHub 사용법 


### 버전 관리 시스템(VCS)의 필요성

버전 관리 시스템은 프로젝트의 변경 이력을 저장하고 관리하는 기술이다. 개발 과정에서 작성된 코드, 문서, 설정 파일 등은 지속적으로 수정되는데, 이러한 변화가 누적되면서 이전 버전으로 되돌아가야 하는 상황이 자주 발생한다. 대표적인 상황은 다음과 같다.

* 새로운 기능 개발 중 심각한 오류 발생 → 과거 안정 버전으로 되돌리기
* 팀원이 실수로 코드를 삭제하거나 훼손함 → 이전 커밋에서 복구
* 프로젝트가 여러 명에 의해 동시에 작업됨 → 충돌 방지 및 관리 필요

버전 관리 시스템을 사용하지 않고 파일을 직접 복사하는 방식으로 관리하면 불필요한 혼란이 생기며, 협업 시 동일 파일 수정으로 인해 손실이 발생할 수 있다. VCS는 이러한 문제를 해결하기 위해 각 변경점을 ‘버전’으로 저장하고 필요에 따라 비교, 복원, 병합할 수 있는 기능을 제공한다.

Git은 단순한 파일 관리 도구가 아니라, 파일의 변경 이력을 스냅샷(snapshot) 형태로 저장하고 관리하는 장치다. Git 내부 구조는 크게 다음 세 가지 영역으로 구성.

 Working Directory

* 실제 파일을 수정하는 작업 영역.
* 현재 체크아웃된 브랜치의 최신 버전이 위치함.

  Staging Area (Index)

* 커밋하기 전 변경된 파일을 임시로 저장하는 공간.
* “이번 커밋에 포함할 파일을 선택하는 단계”라고 볼 수 있음.

  Local Repository

* 모든 버전 이력이 저장되는 영역.
* 스냅샷 형태로 커밋 기록이 보관되며, 필요 시 과거 버전으로 되돌릴 수 있음.

### Git의 장점과 활용 사례

Git은 다음과 같은 이유로 개발자들에게 필수 도구가 되었다.

* **빠른 속도**: 로컬에서 대부분의 작업을 수행하므로 commit, branch 생성 등이 매우 빠르다.
* **강력한 브랜치 기능**: 기능 개발, 버그 수정, 실험 등을 독립적인 브랜치에서 수행 가능.
* **충돌 해결 능력**: 동일 파일을 여러 사람이 수정하더라도 비교·병합 기능이 뛰어남.
* **오프라인 작업 가능**: 네트워크 접속 없이도 전체 히스토리 조회 및 작업 가능.
* **안정성과 신뢰성**: 스냅샷 기반 저장 방식으로 데이터 유실 위험이 적고 무결성이 보장됨.

## Git 설치 및 초기 설정

### 사용자 정보 설정

Git은 커밋을 작성할 때 작성자 정보를 기록하므로, 누구의 커밋인지 추적하기 위해 아래 설정이 필요하다.

#### 사용자 이름 설정

```
git config --global user.name "사용자이름"
```

#### 사용자 이메일 설정

```
git config --global user.email "이메일주소"
```

예시:

```
git config --global user.name "ksb"
git config --global user.email "example@email.com"
```

GitHub 계정에 등록된 이메일과 동일할 필요는 없지만, GitHub 연동 시 동일하게 설정하면 기여 내역이 정확히 표시된다.

### 기본 환경 설정

Git은 다양한 설정을 제공하며 필요에 따라 환경을 개선할 수 있다.

#### 1) 기본 에디터 설정

Git은 커밋 메시지 작성 시 기본 에디터를 실행하는데, 편한 에디터를 설정할 수 있다.

```
git config --global core.editor "code --wait"
```

이 명령은 VSCode를 기본 Git 에디터로 지정한다.



#### 2) 자주 사용하는 설정 확인 방법

설정값을 확인하려면 다음 명령어를 사용한다.

```
git config --list
```

### Git 초기화 및 저장소 생성

Git을 실제로 사용하기 위해서는 프로젝트 디렉토리를 Git 저장소로 초기화해야 한다.

#### 로컬 저장소 생성

```
git init
```

이 명령을 실행하면 `.git` 폴더가 생성되고 해당 폴더가 버전 관리의 핵심 역할을 한다.

#### 기존 프로젝트를 Git으로 관리하기

기존 프로젝트에서 Git을 적용하려면 프로젝트 폴더에서 단순히 `git init`을 실행하면 된다.

### 3.6 Git 상태 확인과 기본 흐름 이해

Git을 설치하고 초기 설정까지 완료했다면 가장 먼저 알아야 할 명령어는 다음 두 가지이다.

####  상태 확인

```
git status
```

* 변경된 파일
* 스테이징된 파일
* 커밋하지 않은 변경 사항
* 현재 브랜치
  등을 확인할 수 있다.

####  파일 추가 및 커밋 흐름

Git의 기본 흐름은 다음과 같다.

1. 파일 수정 → Working Directory
2. 파일 선택 → Staging Area (`git add`)
3. 커밋 생성 → Repository (`git commit`)

간단한 예시는 다음과 같다.

```
git add .
git commit -m "첫 커밋"
```

Git 설치와 초기 설정이 완료되면 실제로 Git 기능을 본격적으로 사용할 준비가 끝난 것이다. 다음 섹션에서는 Git의 가장 핵심이 되는 명령어들을 보다 깊이 있게 다룬다.

## Git의 핵심 개념과 명령어

Git을 실제로 활용하기 위해 반드시 이해해야 하는 요소는 저장소, 스테이징, 커밋, 브랜치, 병합 등 핵심 개념과 그에 대응하는 명령어들이다. Git은 강력하지만 단순한 구조로 이루어져 있어 개념만 정확히 이해하면 효율적이고 안정적인 버전 관리를 수행할 수 있다.

###  저장소(Repository)

저장소는 Git이 변경 이력을 관리하는 공간이다. 저장소는 두 종류로 나뉜다.

* **로컬 저장소(Local Repository)**: 개발자의 PC에 위치한 저장소.
* **리모트 저장소(Remote Repository)**: GitHub 같은 서버에 위치한 저장소.

로컬 저장소 생성:

```
git init
```

리모트 저장소 연결:

```
git remote add origin 저장소주소
```

###  스테이징(Staging Area)

스테이징 영역은 "커밋할 파일을 선택하는 공간"이다. Working Directory에서 수정된 파일 중 이번 커밋에 포함시킬 파일을 Staging Area에 올리는 과정을 staging이라 한다.

모든 파일을 스테이지에 추가:

```
git add .
```

특정 파일만 추가:

```
git add 파일명
```

스테이징 상태 확인:

```
git status
```

###  커밋(Commit)

커밋은 프로젝트의 특정 시점을 기록한 스냅샷이다. Git의 커밋은 SHA-1 해시값으로 구분되며, 모든 커밋은 부모 커밋과 연결되어 프로젝트의 전체 히스토리를 구성한다.

커밋 생성:

```
git commit -m "메시지"
```

이전 커밋 로그 확인:

```
git log
```

한 줄 요약 형태의 로그:

```
git log --oneline
```

###  브랜치(Branch)

브랜치는 Git의 가장 핵심 기능 중 하나로, 독립적인 작업 공간을 의미한다. 새로운 기능 개발, 실험, 버그 수정 등 다양한 작업을 메인 코드와 분리해 진행할 수 있도록 한다.

브랜치 목록 확인:

```
git branch
```

새 브랜치 생성:

```
git branch 브랜치명
```

브랜치 이동:

```
git checkout 브랜치명
```

브랜치 생성과 이동을 동시에:

```
git checkout -b 브랜치명
```

###  병합(Merge)

병합은 한 브랜치의 변경 사항을 다른 브랜치에 통합하는 과정이다. 일반적으로 기능 개발 브랜치를 완성한 main 브랜치로 병합한다.

병합 예시 (feature → main):

```
git checkout main
git merge feature
```

병합 시 충돌이 발생하는 경우 Git은 수동 해결을 요구하며, 충돌 해결 후 다시 커밋해야 병합이 완료된다.

###  되돌리기(Reset, Revert)

Git에서는 실수한 내용을 되돌리는 방법이 다양하다.

####  revert(안전한 되돌리기)

특정 커밋의 변경 내용을 취소하지만 히스토리는 보존한다.

```
git revert 커밋ID
```

협업 환경에서 가장 안전한 방법이다.

####  reset(기록 자체를 이전 시점으로 삭제)

이전 커밋 상태로 기록이 돌아간다.

```
git reset --hard 커밋ID
```



###  push, pull, fetch

로컬 작업 결과를 리모트 저장소에 반영하거나, 리모트 변경 내용을 로컬로 가져올 때 사용하는 명령어들이다.

#### push (로컬 → 리모트 업로드)

```
git push origin 브랜치명
```

#### pull (리모트 변경 내용 가져오고 자동 병합)

```
git pull origin 브랜치명
```

#### fetch (리모트 변경 내용 가져오지만 병합은 하지 않음)

```
git fetch
```

fetch 후 병합은 개발자가 직접 수행한다. 이를 통해 충돌을 미리 확인하거나 반영 여부를 선택할 수 있다.

### 4.8 Git의 기본 워크플로우

Git을 사용할 때 가장 일반적인 작업 흐름.

1. 코드 수정 → Working Directory
2. 수정된 파일 선택 → `git add`
3. 변경 내용을 저장 → `git commit`
4. 리모트 저장소에 반영 → `git push`
5. 협업자의 작업 내용 가져오기 → `git pull`



## GitHub란 무엇인가

GitHub는 Git 저장소를 온라인에서 관리하고 협업을 가능하게 하는 세계 최대 규모의 소스 코드 호스팅 플랫폼이다. Git을 단독으로 사용하면 로컬 저장소에서 버전 관리는 가능하지만, 여러 개발자와의 협업이나 코드 공유는 쉽지 않다. GitHub는 이러한 Git의 한계를 확장하여 코드 저장, 협업, 리뷰, 자동화, 배포까지 지원하는 통합 개발 환경을 제공한다.


###  GitHub의 정의

GitHub는 Git 기반 프로젝트를 클라우드 서버에 저장하고 관리할 수 있는 플랫폼이다. GitHub는 웹과 앱 형태의 인터페이스를 제공하여 사용자가 명령어 없이도 저장소를 관리할 수 있도록 돕는다. 

* Git 기반 리모트 저장소 제공
* 협업 기능 (Pull Request, Issues, Projects)
* 팀 관리 기능 (권한, 조직, 팀 분리)
* 코드 리뷰 관리
* 소프트웨어 배포 및 릴리스 관리
* 워크플로우 자동화(GitHub Actions)
* 오픈소스 커뮤니티 기능

GitHub는 Git을 기반으로 하지만 Git과는 별개인 서비스 플랫폼이다. Git은 버전 관리 도구이고, GitHub는 Git을 사용하는 프로젝트들이 함께 공유하고 관리할 수 있는 “온라인 공간”이다.


### GitHub의 주요 기능


####  저장소(Repository)

GitHub의 핵심 요소로, 프로젝트 파일과 커밋 이력, 브랜치 등이 포함된다. 저장소는 Public 또는 Private으로 설정할 수 있다.

####  Issues(이슈 추적 시스템)

프로젝트 진행 중 발생하는 작업 항목, 버그, 질문 등을 관리하는 시스템이다.

* 작업 단위 정의
* 버그 보고
* 기능 제안
* 개발 일정 관리

라벨, 마일스톤과 결합하면 프로젝트 관리 도구로 활용할 수 있다.

####  Pull Request (PR)

PR은 GitHub 협업의 핵심이다. 한 개발자가 브랜치에서 작업한 코드를 다른 브랜치(main 등)에 병합해 달라고 요청하는 기능이다. PR은 다음 작업을 포함한다.

* 코드 리뷰
* 변경 파일 비교
* 자동 머지 또는 충돌 확인
* 코멘트 기반 협업
* 승인(Approve) 후 병합

PR을 통해 코드 품질을 높이고 프로젝트의 안정성을 유지할 수 있다.

####

GitHub Issues와 연동해 프로젝트 전체를 시각적으로 관리할 수 있다.

###  GitHub에서 사용하는 기본 용어

GitHub를 사용할 때 자주 등장하는 용어를 이해하면 전체 흐름을 쉽게 파악할 수 있다.

* **Repository**: 프로젝트 저장소
* **Branch**: 기능 단위 작업 공간
* **Commit**: 변경 이력 단위
* **Pull Request**: 브랜치를 병합해 달라는 요청
* **Fork**: 저장소 복제(개인 계정으로 가져오기)
* **Star**: 관심 저장소 표시
* **Clone**: 저장소를 로컬로 다운로드하는 작업


##  Git과 GitHub 연동하기

Git과 GitHub를 연동하면 로컬에서 작업한 내용을 GitHub 원격 저장소에 안전하게 업로드하고, 협업자들과 변경 사항을 공유하며, 팀 개발 환경을 효율적으로 구성할 수 있다. 이 섹션에서는 GitHub 저장소 생성부터 리모트 연결, push/pull 작업, 인증 방식까지 모든 단계를 상세히 설명한다.

###  GitHub 저장소 생성하기

GitHub에서 새로운 저장소를 만들기 위해 다음 과정을 따른다.

####  GitHub 로그인 후 Repositories → New 클릭

GitHub 오른쪽 상단 프로필 메뉴에서 **Your repositories**를 선택하고, **New** 버튼을 클릭한다.

####  저장소 이름 입력

예: `my-project`

#### 3) Public(공개) 또는 Private(비공개) 선택

* **Public**: 누구나 접근 가능
* **Private**: 초대된 사용자만 접근 가능

#### 4) README 생성 여부 선택

처음 생성 시 README를 함께 만들면 기본 구조가 자동으로 생성되므로 편리하다.

#### 5) Create Repository 클릭

새 저장소가 생성되며, HTTPS 또는 SSH 방식의 저장소 주소를 확인할 수 있다.

### 6.2 로컬 프로젝트와 GitHub 저장소 연결하기

GitHub 저장소와 로컬 저장소를 연동하려면 Git 리모트를 추가해야 한다.

####  기존 프로젝트에 Git 저장소 초기화

```
git init
```

####  GitHub 리모트 저장소 주소 등록

GitHub에서 제공하는 URL을 복사해 다음 명령어를 입력한다.

```
git remote add origin https://github.com/사용자명/저장소명.git
```

리모트 목록 확인:

```
git remote -v
```

###  GitHub로 push 하기

로컬 커밋을 GitHub에 업로드하려면 다음 단계를 진행한다.

####  작업 파일 스테이징

```
git add .
```

####  커밋 생성

```
git commit -m "첫 커밋"
```

####  main 브랜치로 push

GitHub 기본 브랜치 이름은 `main`인 경우가 많다.

```
git push -u origin main
```

`-u` 옵션은 앞으로 push/pull 시 자동으로 해당 리모트를 기준으로 사용하도록 설정한다.

###  GitHub에서 pull 하기

협업자가 새로운 변경 사항을 push했다면, 이를 로컬에 가져와야 한다.

```
git pull origin main
```

pull은 **fetch + merge** 기능을 동시에 수행하므로 자동으로 병합까지 처리된다.


###  GitHub 저장소 클론(clone)하기

다른 사람이 만든 저장소 또는 팀 저장소를 로컬에 복사하려면 clone을 사용한다.

```
git clone https://github.com/사용자명/저장소명.git
```

clone을 사용하면 다음이 자동으로 설정된다.

* 로컬 저장소 생성
* 원격 리모트(origin) 자동 연결
* 최신 커밋 다운로드

###  로컬과 원격 저장소의 관계 이해하기

Git은 로컬과 원격 저장소를 독립적으로 관리한다. 다음과 같은 관계로 구성된다.

| 구분      | 로컬 저장소  | 원격 저장소             |
| ------- | ------- | ------------------ |
| 위치      | 개인 PC   | GitHub 서버          |
| 접근 방식   | Git 명령어 | GitHub 웹 UI 또는 Git |
| 업데이트 방식 | commit  | push / pull        |

로컬에서 commit 한 후, push해야만 GitHub에 반영된다.
